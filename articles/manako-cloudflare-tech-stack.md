個人開発でSaaSを作ると、ユーザーが0人でもインフラ費が月数千円かかる。AWS RDSだけで$15。時系列DBを足したら$45。まだ1円も売上がないのに。

**もし月$5で全部動いたら?** Cloudflareスタックならそれができます。

この記事では、監視SaaSの技術選定で「Cloudflare全部のせ構成」を選んだ理由と、設計上のトレードオフを正直に書きます。

## 従来構成 vs Cloudflare構成

```
AWS構成:   Lambda + RDS + Timestream + SQS + ElastiCache = $100~200/月
Cloudflare構成: Workers + D1 + Analytics Engine + Queues + KV = $5/月

→ 95%以上のコスト削減
```

| 項目 | Cloudflare | AWS | Vercel + Supabase |
|------|-----------|-----|-------------------|
| API サーバー | Workers ($5/mo paid) | Lambda (無料枠) | Functions (無料枠) |
| DB | D1 (5GB無料) | RDS ($15~/mo) | Supabase (500MB無料) |
| Cron 実行 | Cron Triggers (無料) | EventBridge + Lambda | Cron (1日1回) |
| キャッシュ | KV (無料枠あり) | ElastiCache ($15~/mo) | Redis ($0~) |
| 時系列データ | Analytics Engine (無料) | Timestream ($$$) | TimescaleDB ($30~/mo) |
| キュー | Queues (無料枠あり) | SQS (従量) | 別途必要 |
| **月額(初期)** | **$5** | **$50~100** | **$0~25** |
| **月額(100ユーザー)** | **$5~10** | **$100~200** | **$50~80** |

ただし当然トレードオフがある。数字の裏にある設計判断が本題です。

## 最大の設計判断: 時系列データを D1 に入れない

監視SaaSで最もデータ量が多いのは **チェック結果** です。100個のモニターを1分間隔でチェックすると、月に約432万レコード。

これを D1(SQLite)に入れるとどうなるか。D1 のコスト試算をした結果がこうでした:

| ユーザー数 | D1 writes/月 | D1 超過コスト | Analytics Engine |
|-----------|-------------|-------------|-----------------|
| 100 | 60.5M | $10.5/月 | ほぼ無料 |
| 200 | 172.8M | **$122.8/月** | $19/月 |
| 500 | 432M | $382/月 | $47/月 |

**200ユーザーで月$122 vs $19。** この差を見て、チェック結果は D1 に入れないと決めました。

D1 は公式にも read-heavy ワークロードに最適化されている、と言っています。監視の時系列書き込みは明確に向いていない。

## Analytics Engine のトレードオフ

ただし Analytics Engine は万能ではないです。受け入れたトレードオフを正直に書きます。

**1. クエリが遅い**

D1 はローカルクエリなので数ms。一方 Analytics Engine は GraphQL API 経由で数百ms かかります。ダッシュボードの表示速度に影響する。ただし監視ダッシュボードの更新頻度は1分単位なので、数百ms の遅延は許容できると判断しました。

**2. サンプリングが入る場合がある**

大量データの集計時に Cloudflare がサンプリングを行うことがあります。つまり100%正確な数値ではない可能性がある。監視SaaSで「稼働率99.98%」と表示して、実は99.97%だったら困るのでは? -- 正直、個人開発者向けの価格帯でこの精度を気にするユーザーはほぼいないと割り切りました。

**3. 保持期間が最大3ヶ月**

Analytics Engine のデータ保持は最大90日。最初は制約に感じましたが、考えてみると **チェック結果は、アラートさえ飛べば役目を果たしている** データです。異常を検知して通知した時点で価値の大部分は消化されていて、3ヶ月前のレスポンスタイムを見返す場面はほぼない。

長期保持が必要なのは監査ログ(誰が何をしたか)やインシデント履歴であって、これらは D1 に入れています。チェック結果の保持期間と混同しないことで、設計がシンプルになりました。

**4. 個別チェック結果の詳細ページは作らない**

Analytics Engine は集計クエリに最適化されているので、「このチェックのこの瞬間のレスポンスボディ」みたいな詳細は引けません。ダッシュボードは集計ビュー(稼働率、レスポンスタイム推移)に絞る設計にしています。

## D1 には何を入れるか

D1 に入れるのは **設定データと状態データ** だけです。

- **設定**: モニター定義、チーム、ユーザー、通知チャンネル、APIキー、ステータスページ
- **状態**: モニターの現在ステータス、最終チェック日時、インシデント、監査ログ

全24テーブル。監視タイプは HTTP、Heartbeat、WebChange、TCP、SSL、Domain、Ping の7種類に対応しています。

書き込み頻度が低いものだけ。チェックのたびに更新するのは `lastCheckedAt`(毎回)と `status`(変化時のみ)の2カラムだけで、これなら200ユーザーでも D1 の無料枠に収まります。

D1 はマルチテナントで使うので、主要テーブルに `team_id` カラムを持たせて全クエリに `WHERE team_id = ?` を付ける行レベル分離を採用しています。24テーブル中15テーブルが直接 `team_id` を持ち、残りの9テーブル(中間テーブルやスナップショットなど)は親テーブル経由で間接的にテナント分離されます。D1 の基盤は SQLite なので RLS(Row Level Security)が使えず、テナント分離はアプリケーション側の責任。シンプルだけど、1箇所書き忘れるとデータ漏洩になるので、レビューとミドルウェアで防ぐ運用にしています。

**役割分担をまとめると:**

| データ | 保存先 | 理由 |
|--------|--------|------|
| 設定・状態 | D1 | Read-heavy。strong consistency が必要。team_id で行レベル分離 |
| チェック結果(時系列) | Analytics Engine | Write-heavy。集計クエリ向き |
| セッション・キャッシュ | KV | TTL で自動削除。揮発性 OK |
| 通知 | Queues | 非同期処理。リトライ付き |

## Cron Triggers: 1分間隔の設計

監視SaaSの心臓部は定期チェックです。Vercel の Cron は無料枠だと1日1回で、監視の用途には合いませんでした。

Workers の Cron Triggers は1分間隔まで設定可能で、追加料金なし。ただし設計上の注意点がありました。

**全モニターを毎分チェックするわけではない。** モニターごとに間隔が違います(1分、5分、15分など)。毎分 Cron が起動して、「今チェックすべきモニター」を D1 から SQL で抽出し、該当するものだけ実行する設計です。

**Workers の CPU 時間制限**(30秒)があるので、1回の Cron でチェックできるモニター数に上限があります。計算上、約800ユーザー(モニター数 数千)で限界に達する見込み。そこから先は Queue で checker Worker に分散する設計に切り替える予定ですが、MVP では単一 Worker で十分です。

## インフラ費を気にしなくていい構成

個人開発で一番怖いのは「ユーザーが来なくてもインフラ費がかかり続ける」こと。

Workers Paid Plan **月額 $5** で、DB・KV・Queue・Analytics Engine・Cron Triggers が全部含まれている。ドメイン代を合わせても年間 $70 程度。ユーザーが100人に増えても、インフラ費は $5~10 のまま。

ユーザーが0人でも月 $5 なら痛くないし、増えても比例して膨らまない。個人開発で SaaS を続けるには、この「怖くなさ」が大事だと思っています。

## まとめ

- 個人開発SaaSのインフラは **月$5** から始められる
- 最大の設計判断は **時系列データを D1 に入れない** こと。200ユーザーで月$122 vs $19の差が出る
- Analytics Engine はトレードオフがある(クエリ遅い、サンプリング、90日保持)。割り切れるなら強い選択肢
- D1 は設定・状態データに絞り、write を最小化するのがコツ
- ユーザーが0人でも100人でも、インフラ費がほぼ変わらない「怖くない構成」が個人開発には大事
