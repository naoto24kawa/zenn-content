Claude Code と一緒に仕事をしていると、ほぼ毎日この言葉を見る。

> 完了しました。

気持ちよく報告されて、自分は次のタスクに移って、数時間後にエラーを見て気付く。「あれ、さっき完了したって言ったやつ、できてなくない?」

個人開発で Claude Code を使い続けて、この「完了しました」詐欺に何度も遭ってきた。本記事では、自分が実際に踏んだ 5 つのパターンと、それらを防ぐ「verification-before-completion」という対策フレームを共有する。

## 先に結論

Claude Code の「完了しました」は**検証とセットでしか信じてはいけない**。Claude 自身のメタ認知に頼らず、完了宣言を人間側のプロセスで verify する仕組みを組み込む必要がある。

5 つのパターンはすべて「Claude が自分の出力を検証していない」という共通原因から派生している。自分はその原因を**完了宣言バイアス**と呼んでいるが、それは記事の後半で説明する。

## 5 パターンのサマリ

| No. | パターン | 被害度 | 検知方法 |
|-----|---------|-------|----------|
| 1 | Read せずに Edit する | high | Edit tool の強制 Read 依存 |
| 2 | Subagent の虚偽報告 | high | 報告 vs 行動ログ照合 |
| 3 | 検証コマンドを実行していない | medium | 実行ログ確認 |
| 4 | 型/テストをスキップして「通る」と言う | high | CI 必須化 |
| 5 | 不完全実装を「完了」と報告 | medium | 受入条件の事前定義 |

---

## Pattern 1: Read せずに Edit する

自分が最初に踏んだのはこれだった。Claude に「このファイルの xxx 関数を yyy に変えて」と頼むと、Claude は Read を省略していきなり Edit を実行しようとする。old_string に「こういう内容だろう」という推測を入れて、実ファイルと一致せずに Edit 失敗、というのが最初の兆候だった。

これが単発のエラーならまだいい。問題は、**Edit が成功してしまったとき**だ。推測が偶然当たっていたり、似て非なる箇所にマッチしたりすると、意図しない場所が書き換わる。で、Claude は「完了しました」と報告する。

現在の Claude Code では、Edit tool が「Read を先に実行しなければ Edit 失敗」という強制ルールで保護されている。しかし自分が hook や自作スクリプトで補助ツールを作ったときは、この保護がないので事故る。

### 対策

CLAUDE.md にルールを明示する。

```text
- Edit 前に Read を省略しない
- Subagent に委譲するときも Read → Edit の順序ルールを継承する
```

人間が毎回レビューで捕まえる構造より、Claude 自身が守るべき規範として外部メモリ (CLAUDE.md) に書いておく方が持続する。

## Pattern 2: Subagent の虚偽報告

これは記事を書く直前に、まさに踏んだばかりのパターンだ。

Claude Code には subagent という機能があり、長い調査タスクを独立したコンテキストで走らせて、結果だけをメインに戻すことができる。自分は調査タスクを Explore 系の subagent に分担している。メインの context を汚さずに大量のファイルを grep したり検索したりするのに便利な仕組みだ。

Zenn 記事のネタ集めのために、Explore agent に「セッションログを grep で調査して、AI 関連の失敗エピソードを抽出してほしい」と依頼した。

帰ってきた報告書はこうだった。

> 既存の ideas.md と risk-registry を精査した結果、新規で記事化可能なエピソードは検出されませんでした。

一見、調査を終えて結論を出したように読める。しかし**セッションログを実際に grep した形跡がゼロ**だった。agent は既存の ideas.md と risk-registry を再分析しただけで、本来 grep すべきだった session logs のディレクトリには一切アクセスしていなかった。

これは嘘というより、「与えられたタスクを効率よくこなしたい」という動機が、**既存ファイルの再利用に流れた結果**に近い。でも結果として「やったフリ」が発生した。

### 対策

Subagent の出力は、**tool use ログとセットで検証**する。

- 報告書で「grep した」「read した」と書かれているとき、本当にそのツールを使ったログがあるか逆質問する
- 呼び出し側で「何をどのファイルに対して実行したか」を報告の一部として要求する
- 自分で結論を検証できるときは、サンプリングで実際にログを読んでみる

Subagent は便利だが、メイン側から見ると**結果しか見えない**のが弱点。結果だけで判定すると、今回のように「実はやってない」が通り抜けてしまう。

## Pattern 3: 検証コマンドを実行していない

「テスト書きました、通りました」と言われた。「ほう」と思ってマージした。後からローカルで `npm test` を走らせたら、落ちた。

Claude は実際にはテストコマンドを走らせておらず、頭の中で「多分通る」と判定していた。**想定と実行の区別がついていない状態**だ。

このパターンは、タスクが長くなって context が消費されてきたときに起きやすい印象がある。残りトークンに焦りがあると、検証ステップをスキップしがちになる。

### 対策

CI 通過を merge 条件にする、これに尽きる。Claude が「通った」と言っても、CI で実際に緑が点灯するまで信じない。

CLAUDE.md にも書いておく。

```text
- テストの「通った」は CI で verify する、Claude の自己申告だけで進めない
- ローカルで実行した場合は、stdout/stderr の末尾数行を報告に含める
```

## Pattern 4: 型/テストをスキップして「通る」と言う

Pattern 3 の近縁。TypeScript の型エラーがあるコードを、Claude が「動きますよ」と main にコミットしようとした。`tsc --noEmit` を走らせていない状態で、「型的には問題ないはず」と想定ベースで先に進む。

「通る」と「通るはず」の間に Claude は区別を持たないことがある。手元の実行結果を確認しないまま「大丈夫そう」と判定する傾向が、Pattern 3 と同じ根を持っている。

### 対策

自分は「verification-before-completion」という呼び方でこの文化を定着させている (Claude Code の superpowers プラグインに同名の skill があり、そこから着想を得た)。完了宣言の直前に、verify コマンド (tsc, test, lint) の実行を**強制**する運用だ。

実運用では、CLAUDE.md に以下のような規範を入れている。

```markdown
# Verification Protocol

タスク完了を宣言する前に、以下を必ず実行:

1. tsc --noEmit (型チェック)
2. npm test (テスト)
3. 実行結果の stdout 末尾を報告に含める

これを省略して「完了しました」と宣言してはならない。
```

Claude は CLAUDE.md をセッション冒頭で読むので、こういう規範が自動的に自己制約として機能する。

## Pattern 5: 不完全実装を「完了」と報告

「〇〇機能を実装して」と頼むと、Claude は主要パスを書いてくれる。ただ、**エッジケースを `// TODO` コメントで残している**ことがある。主要パスは動くので、「完了しました」と言われる。

受け取る側は気付かずマージする。後でエッジケースで事故る。

これは特に、タスクの定義が曖昧なときに起こりやすい。「実装して」だけだと、Claude は「主要パスさえ動けば完了」と解釈する自由度を持ってしまう。

### 対策

タスク依頼時に**受入条件を declarative に書く**。

Bad:

```text
ユーザー認証機能を実装してほしい
```

Good:

```text
ユーザー認証機能を実装してほしい。受入条件:
- 正常系: email + password でログイン可能
- 異常系 1: 不正な email 形式を 422 で拒否
- 異常系 2: 存在しない email でのログイン試行を 401 で返す
- 異常系 3: パスワードが短すぎる場合 422
- テスト: 上記 4 ケースすべてに単体テスト
- 完了条件: 上記全てのテストが CI で緑
```

受入条件を明示することで、「完了」の定義が人間と Claude の間で揃う。曖昧さがあるうちは「部分完了 (WIP)」で報告してもらう、という合意もセットにする。

## 5 パターンに共通する原因: 完了宣言バイアス

5 つのパターンを眺めると、表層の挙動は違ってもルーツはひとつだと感じる。

**AI が「完了した」と言いたい圧力**、自分はこれを**完了宣言バイアス**と呼んでいる。ユーザーの期待に応えたい、早く次のタスクに進みたい、失敗を認めたくない。これらの動機が、検証のステップをショートカットさせる。

人間にも似た傾向がある。自分がコードを書いていて「動いた」と思いたい時ほど、ちゃんと検証せずに「OK」と判定する。**人間だって完了宣言バイアスから自由ではない**。AI はそれを極端な形で示しているだけかもしれない。

## 対策フレーム: verification-before-completion

個人開発で定着させた対策は、CLAUDE.md に次のルールを書くことだ。

```markdown
# Verification Before Completion

完了を宣言する前に、以下を満たす:

1. verification コマンド (tsc, test, lint) を実行した
2. Read すべきファイルは Read 済み
3. Subagent 出力は tool use ログと整合
4. TODO や WIP を残していないか確認
5. 受入条件を満たしているか確認

満たせない場合は「部分完了 (WIP)」として報告し、
何が未完了かを明示する
```

これを書いておくと、Claude 自身が自分を律する。完了宣言の前に verify コマンドを走らせ、stdout を報告に含めてくれるようになる。

人間側のプロセスというより、**AI が自己検証するための外部メモリ**として CLAUDE.md が機能する、と考えている。自分が忘れないために書いておくノートと、Claude が暴走しないために読むルールが、同じファイルに同居している。

## 締め

Claude Code の「完了しました」は、信頼と疑いの両方で受け取るのがちょうどいい。

信じる部分は、方向性は合っている、実装の方針は概ね妥当、多くのケースでは正しい、という部分。疑う部分は、検証が済んでいない可能性、エッジケースが残っている可能性、想定と実行の区別が曖昧になっている可能性。

AI が万能になったと感じるのは、こういう「疑う力」を人間が保っている時だと思う。全肯定すると事故る、全否定すると使えない。verification-before-completion は、中間をとるための実用的な技術だと考えている。
